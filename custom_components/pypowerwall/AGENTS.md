# AGENTS.md — custom_components/pypowerwall/

Detailed, file-by-file rationale for this integration's implementation. See the [root AGENTS.md](../../AGENTS.md) for project-wide commands, release process, and conventions.

## Files

- `manifest.json` — HA metadata; declares `pypowerwall` as the only runtime requirement (HA installs it itself from this, independent of `requirements.txt`).
- `const.py` — `DOMAIN`, config keys, the six `CONN_TYPE_*` connection-mode identifiers, default scan interval.
- `config_flow.py` — `async_step_user` shows a menu (`async_show_menu`) of the six connection types; each type has its own `async_step_<conn_type>` (method name == `CONN_TYPE_*` value, since HA routes a menu selection to `async_step_<next_step_id>`) with its own voluptuous schema, all funneling through the shared `_async_step_connection()` which validates the connection and uses the gateway's DIN as the config entry's `unique_id` (so re-adding the same gateway aborts as "already configured"). Also an options flow for scan interval. Cloud/FleetAPI/TEDAPI-v1r forms take a filesystem path rather than credentials — see README's connection-type table for why (Tesla's OAuth login can't be driven from a single form submission).
- `coordinator.py` — `PowerwallDataUpdateCoordinator`, a `DataUpdateCoordinator` that polls a synchronous `pypowerwall.Powerwall` client via `hass.async_add_executor_job` (pypowerwall has no async API) and maps its methods onto a `PowerwallData` dataclass. `build_powerwall_kwargs(conn_type, data)` translates a config entry's `data` dict into the right `pypowerwall.Powerwall(**kwargs)` call shape for each of the six modes — this is the single place that knows what each mode needs; both `config_flow.py` (to validate) and `async_connect_powerwall()` (to actually connect on setup) call through it, so adding a mode only means touching this function once. `async_connect_powerwall()` does the initial connect + `is_connected()` check, raising `ConfigEntryNotReady` on failure so HA retries setup. `_fetch_data()` calls `pw.battery(verbose=True)` (not the plain `pw.battery()` used for the other power readings) to get the full `/api/meters/aggregates` battery-meter dict rather than just `instant_power` — that same dict carries the gateway's own lifetime `energy_imported`/`energy_exported` counters (Wh), which `battery_energy_imported`/`battery_energy_exported` convert to kWh via `_wh_to_kwh()`. This is the actual hardware meter reading, not a client-side power integration, and it's available uniformly across every connection mode (verified against pypowerwall's local/cloud/fleetapi/tedapi backends, which all synthesize this same aggregates payload).
- `entity.py` — `PowerwallEntity` base class; builds shared `device_info` (one HA device per gateway, keyed by DIN) so every entity below groups under it.
- `sensor.py` — static sensors declared as `PowerwallSensorDescription` (a `SensorEntityDescription` + `value_fn`), plus per-battery-pack temperature sensors added dynamically as new devices show up in `coordinator.data.temps` (tracked via a `known_temp_devices` set and a coordinator listener, since the gateway doesn't report how many battery packs exist up front). Deliberately has no `battery_reserve`/`battery_mode` entries — those are writable, so they live in `number.py`/`select.py` instead of being duplicated as read-only sensors. `battery_energy_imported`/`battery_energy_exported` are `device_class: energy`, `state_class: total_increasing` sensors meant to be selected directly in HA's Energy dashboard battery config — see `coordinator.py`'s note on where the underlying data comes from. `battery_import_power`/`battery_export_power` are pure derivations of `data.battery_power` (module-level `_battery_import_power()`/`_battery_export_power()` functions, not lambdas, since the logic is more than a one-line field lookup) rather than new coordinator fields — same precedent as `alert_count`'s `len(data.alerts)`. Sign convention, confirmed by working through the power balance in pypowerwall's own example `/api/meters/aggregates` payload (README.md's dump around line 828 in the `pypowerwall` repo): **negative `battery_power` = charging (import), positive = discharging (export)** — matches `battery_energy_imported`/`_exported`'s charge/discharge mapping, so don't flip the sign if you touch this. `grid_import_power`/`grid_export_power` (`_grid_import_power()`/`_grid_export_power()`) are the same derivation pattern applied to `data.grid_power`, but with the **opposite** sign convention from battery: per pypowerwall's own README glossary (`energy_imported` = "kWh pulled from grid", `energy_exported` = "kWh pushed to grid"), and confirmed by the same example payload's power balance, **positive `grid_power` = importing, negative = exporting** — the reverse of `battery_power`'s charge/discharge sign. Don't copy battery's polarity onto grid; the two are mirrored, not identical. `uptime` has no `native_unit_of_measurement` despite the underlying pypowerwall field being named `up_time_seconds` — it's actually a Go-style duration string (e.g. `"62h48m24.076725628s"`), not a numeric seconds count, and declaring any unit on a sensor makes Home Assistant expect a numeric value regardless of `device_class`/`state_class` (`homeassistant/components/sensor/__init__.py`'s `_numeric_state_expected()`), which raised (caught, logged as an error) on every refresh before this was removed.

  `PowerwallEnergyIntegrationSensor` (also in `sensor.py`) is a client-side Riemann-sum (trapezoidal) fallback for `battery_energy_imported`/`_exported`, for hardware where those gateway-meter fields don't populate reliably. It wires five instances in `async_setup_entry` — `battery_import_energy`, `battery_export_energy`, `grid_import_energy`, `grid_export_energy` (integrating the corresponding `_battery_*`/`_grid_*` power `value_fn`s above), and `home_energy` (integrating `data.home_power` directly, since load has no import/export split) — each given a `source_fn` at construction rather than being five near-duplicate classes. Unlike every other sensor in this file it's stateful (a running total plus the last (power, timestamp) sample, accumulated in `_handle_coordinator_update` using the standalone `_trapezoidal_energy_delta_kwh()` helper) and mixes in `homeassistant.helpers.restore_state.RestoreEntity` to survive HA restarts without resetting to zero — `async_added_to_hass` restores the running total from the last known state, falling back to 0 if there isn't one. A poll where the source reading is `None` is skipped (no delta, previous sample left untouched) rather than treated as a zero-power interval, so the *next* valid reading integrates across the full gap using the last known valid sample. Every instance sets `_attr_entity_registry_enabled_default = False` — deliberately: a client-side power integration is a strictly less accurate approximation of the real hardware meter (missed samples between polls, no sub-interval resolution), so it ships opt-in only rather than silently replacing the real-meter sensors for everyone. Don't flip that default without discussion.
- `binary_sensor.py` — one `connectivity` binary sensor derived from grid status (`UP` → on).
- `number.py` — `PowerwallReserveNumber`, a 0-100% slider wrapping `pw.set_reserve()`.
- `select.py` — `PowerwallModeSelect` wrapping `pw.set_mode()`. `BATTERY_MODES = ["self_consumption", "backup", "autonomous"]` is the exact set pypowerwall's own CLI validates writes against (`pypowerwall/__main__.py`) — not to be confused with Tesla app UI labels like "Backup-Only", which is why `translations/en.json`'s `select.battery_mode.state.*` keys translate the raw values into those friendlier labels for display rather than changing the values themselves. Also `PowerwallGridExportSelect` wrapping `pw.set_grid_export()` — see `switch.py`, it shares that entity's Cloud/FleetAPI gate.
- `switch.py` — `PowerwallGridChargingSwitch` wrapping `pw.set_grid_charging()`. `set_grid_charging`/`set_grid_export` require Cloud or FleetAPI mode per pypowerwall's own docstrings (local/TEDAPI/hybrid backends don't implement them), so this switch and `select.py`'s `PowerwallGridExportSelect` are only added in `async_setup_entry` when `entry.data[CONF_CONN_TYPE]` is in `const.GRID_CONTROL_CONN_TYPES`. The coordinator mirrors the gate: `coordinator._fetch_data()` only polls `get_grid_charging()`/`get_grid_export()` when `conn_type` is one of those two, so non-Cloud/FleetAPI entries don't log an error every poll cycle for a call their backend doesn't support.
- `button.py` — `PowerwallGoOffGridButton`/`PowerwallReconnectGridButton` wrapping `pw.go_off_grid(confirm=True)`/`pw.reconnect_grid()`. As of pypowerwall 0.16.1 neither method is implemented by any backend (`hasattr(self.client, 'go_off_grid')` is always `False` today — it's forward-looking public API surface pypowerwall added ahead of backend support), so unlike the grid-charging/export gate above these buttons are **not** gated by connection type; gating on today's support would just hide them for everyone rather than something meaningful. `PowerwallGoOffGridButton` ships with `_attr_entity_registry_enabled_default = False` since pressing it physically opens the grid contactor (islands the home, ~30s solar dropout) — see README's Entities table callout. A deliberate button press already satisfies pypowerwall's `confirm=True` requirement, so the button always passes it.
- `services.yaml` / `translations/en.json`'s top-level `services` key — `schedule_max_backup`/`cancel_max_backup` wrap `pw.schedule_max_backup()`/`pw.cancel_max_backup()`. Momentary actions rather than persistent state, so they're registered as Home Assistant services (`_async_register_v1r_services()` in `__init__.py`'s `async_setup_entry`) instead of entities, only when `entry.data[CONF_CONN_TYPE] == CONN_TYPE_TEDAPI_V1R` — pypowerwall itself checks `self.tedapi_mode == "v1r"` and no-ops otherwise, and `CONN_TYPE_TEDAPI_V1R` is the only connection type that produces that `tedapi_mode` (see `coordinator.build_powerwall_kwargs()`). Services are domain-scoped rather than per-entry in HA; with more than one v1r entry configured, the most-recently-set-up entry's coordinator ends up handling calls — an accepted limitation given how rare multiple v1r entries would be in practice.
- All write-action platforms (`number.py`, `select.py`, `switch.py`, `button.py`) and the two services follow the same pattern: `await hass.async_add_executor_job(coordinator.pw.<method>, ...)` then `await coordinator.async_request_refresh()` so the entity's state reflects the gateway's actual response rather than assuming the write succeeded.
- `brand/` — `icon.png` (256×256) / `icon@2x.png` (512×512), an original placeholder graphic (not Tesla's logo — custom integrations can't use manufacturer trademarks). HA 2026.3+ reads a custom integration's local `brand/` directory directly and it takes precedence over the [home-assistant/brands](https://github.com/home-assistant/brands) repo, so no external PR is required for this to show up. Swap in real artwork by overwriting these two files at the same names/sizes (square, transparent PNG).
- `translations/en.json` — config flow + entity name text + the `services` block; the sole authoritative English source. **Deliberately no `strings.json`** — that file is a Home Assistant Core-only build-time artifact (Core's build script generates `translations/en.json` from it; custom integrations never run that script), and the [official custom-integration i18n docs](https://developers.home-assistant.io/docs/internationalization/custom_integration/) explicitly warn against shipping one. `translations/<lang>.json` files for other languages (e.g. `fr.json`) can be full copies of `en.json`'s structure, or — since Home Assistant always loads `en.json` first as a fallback and merges the selected language's file on top per-key (`homeassistant/helpers/translation.py`) — a partial file containing only the strings that actually differ (e.g. `en-GB.json` overriding just the handful of AmE/BrE spelling differences). Either way, Home Assistant selects the right file automatically based on the user's configured language; no code changes needed to add one.
